# OrchestrPay - Распределенные платежи

## 📌 Краткое описание
Проектирование и разработка Proof of Concept отказоустойчивой платежной системы с использованием паттерна Saga-оркестрации на базе технологии BPMN-движка. 
Система обеспечивает гарантированную обработку транзакций между B2B/B2C клиентами с автоматическим выполнением компенсирующих операций при сбоях.

## 🎯 Ключевые показатели и цели
* **Reliability**:
  * 100% исключение «зависших» платежей за счет автоматических компенсаций.
  * Контоль качества через архитектурные фитнес-функции и интеграционное тестирование
* **Performance**:
  * SLA < 5 секунд для 90% успешных транзакций.
* **Observability**:
  * Полная визуализация состояния бизнес-процесса в реальном времени через BPMN-мониторинг.  
* **Compliance**: 
  * Интеграция с Anti-fraud и AML системами с поддержкой ручной верификации (Human-in-the-loop).

## 🛠 Технический анализ и Специфика
| **Домен анализа**        | **Характеристика системы**                                                                   |
| ------------------------ | -------------------------------------------------------------------------------------------- |
| **Нагрузка и Scale**     | **Stateful**. Критично сохранение состояния между этапами (списание, фрод, AML).             |
| **Data Consistency**     | **CP (CAP-теорема)**. Строгая согласованность выше доступности.                              |
| **Связность**            | **Distributed**. Необходима независимость доменов (Платежи, Финмониторинг, SRE) .            |
| **Constraints**          | **Compliance**. Требования безопасности, логи аудита и ручные проверки (Human-in-the-loop) . |
| **Эксплуатация**         | **High Observability**. Необходима прозрачность для Support и Security .                     |

## 📐 Архитектурные решения (Key Decisions)

- **Классификация транзакций Saga** (Task 1):
  - Спроектировала реестр шагов, разделив их на компенсируемые (списание), необратимые (pivot point) и повторяемые.
- **Управление состояниями** (Task 2):
  - Разработала таблицу переходов (State Machine), учитывающую сценарии отказа антифрода и тайм-ауты (cut-off time) .
- **Изоляция оркестратора** (Task 3):
  - Подготовила ADR с обоснованием выноса оркестрации в отдельный сервис.
- **Реализация** (Task 4):
  - Спроектировала исполняемую BPMN 2.0 диаграмму в Camunda 8, включая ветку ручной проверки (Human-in-the-loop).
  - Разработала Proof of Concept интеграции BPMN диаграммы с микросервисами через взаимодействие с модульным монолитом (С#).
- **Стратегия тестирования** (Task 5):
  -  Описала план интеграционных и E2E тестов, покрывающих негативные сценарии и дублирующие запросы.
- **Архитектурный комплаенс** (Task 6):
  - Разработала Fitness Functions для автоматического выявления «зависших» Saga и контроля согласованности и дедлайнов платежного бизнес-процесса.

## 📚 Артефакты и Методология
* **BPMN 2.0**: Исполняемые диаграммы процесса обработки платежа.
* **State Machine Table**: Детальное описание переходов и событий системы.
* **Стиль:** Documentation as Code (PlantUML).
* **Визуализация:** Диаграммы в нотации **C4** (Context, Container, Component).
* **Документирование:** **ADR** (Architecture Decision Records) для обоснования выбора технологий.
