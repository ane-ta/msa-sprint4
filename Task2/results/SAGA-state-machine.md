# ADR-\_: Переходы SAGA транзакций

## Контекст

Для управления Saga-оркестрацией необходимо строго детерминировать жизненный цикл платежа . Это гарантирует восстановление процесса после сбоев и корректную отработку компенсаций.

Критичные требования к сценарию платежа:

- Нельзя допустить ситуации, когда деньги зависают между системами.

- Стоимость ошибки при проведении транзакции значительно выше, чем потенциальная потеря от её отклонения.

- Операции возврата должны выполняться автоматически, не привлекая службу поддержки в ручном режиме.

  Если сервисы проверок не возвращают ответ в установленное время, то предусмотрен cut-off-time.

  - По его истечении транзакция по умолчанию считается разрешённой и проводится.

Компенсационные транзакции

- В случае сбоя на любом этапе (например, AML отклонил транзакцию после списания) необходимо
  - инициировать возврат средств
  - и уведомить клиента.

## Проблема

Спроектировать логику переходов состояний (State Machine), которая исключает неопределенность в распределенной транзакции и описывает реакцию на бизнес-события и таймауты.

## Решение

|  |  |  |  |
|----|----|----|----|
| Исходное состояние | Новое состояние | Событие (Триггер) | Описание |
| —————— | —————- | ———————- | ——————————————————- |
| INITIAL | PENDING | ORDER<sub>PLACED</sub> | Клиент подтвердил заказ |
| PENDING | AUTHORIZED | FUNDS<sub>AUTHORIZED</sub> | Средства зарезервированы на счете клиента |
| PENDING | VOIDED | AUTHORIZATION<sub>TIMEOUT</sub> | Клиент не подтвердил оплату в установленное время |
| AUTHORIZED | READY<sub>TO</sub>\_CAPTURE | AUTO<sub>PASS</sub> | Система безопасности признала транзакцию безопасной |
| AUTHORIZED | UNDER<sub>REVIEW</sub> | REVIEW<sub>REQUIRED</sub> | Система безопасности запрашивает ручную проверку |
| AUTHORIZED | PENDING<sub>VOID</sub> | AUTO<sub>REJECT</sub> | Система безопасности признала транзакцию подозрительной |
| AUTHORIZED | READY<sub>TO</sub>\_CAPTURE | VALIDATION<sub>TIMEOUT</sub> | Система безопасности не ответила за установленное время |
| UNDER<sub>REVIEW</sub> | READY<sub>TO</sub>\_CAPTURE | REVIEW<sub>APPROVE</sub> | Ручная проверка признала транзакцию безопасной |
| UNDER<sub>REVIEW</sub> | PENDING<sub>VOID</sub> | REVIEW<sub>REJECT</sub> | Ручная проверка признала транзакцию подозрительной |
| UNDER<sub>REVIEW</sub> | PENDING<sub>VOID</sub> | REVIEW<sub>TIMEOUT</sub> | Ручная проверка не завершилась в установленное время |
| READY<sub>TO</sub>\_CAPTURE | CAPTURED | FUNDS<sub>CAPTURED</sub> | Успех: деньги списаны и получены |
| READY<sub>TO</sub>\_CAPTURE | PENDING<sub>VOID</sub> | PAYMENT<sub>CRITICAL</sub>\_ERROR | Критическая ошибка при переводе денег продавцу |
| PENDING<sub>VOID</sub> | VOIDED | FUNDS<sub>RELEASED</sub> | Резерв денег снят |

## Ограничения и риски

- Риск финансового фрода:
  - Согласно бизнес-правилам, проведение платежа по таймауту (VALIDATION<sub>TIMEOUT</sub>) повышает риск мошенничества, но этот риск признан приемлемым в пользу доступности сервиса.
